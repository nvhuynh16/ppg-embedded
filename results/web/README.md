# Web / career-site assets

Portfolio-ready renderings of the three `ppg-embedded` pipelines. Generated by
the sister scripts `src/make_peak_anim.py`, `src/make_fft_anim.py`, and
`src/make_rr_anim.py` (sharing `src/_anim_helpers.py`) from BIDMC record 01
using the project's own DSP functions, so each visual matches the firmware's
actual computation.

## Files

| File | Size | Pipeline | Use case |
|---|---:|---|---|
| `pipeline.mp4`     | ~5 MB  | Peak detector       | **Primary** — embed with `<video>` on the career site. 32 s loop. |
| `pipeline.gif`     | ~8 MB  | Peak detector       | Fallback — inline `<img>` on GitHub READMEs / Slack / email. 24 s loop. |
| `pipeline_fft.mp4` | ~3 MB  | FFT spectral        | Spectrum + peak-bin + sub-harmonic swap (amber bar when swap fires). |
| `pipeline_fft.gif` | ~3 MB  | FFT spectral        | Same as above, GIF fallback. |
| `pipeline_rr.mp4`  | ~4 MB  | Respiration BW path | Lowpass + 24-bin Goertzel scan + RR readout. |
| `pipeline_rr.gif`  | ~5 MB  | Respiration BW path | GIF fallback. |

**Playback rate (two-knob slowdown):**

| Knob | Where | Default | What it controls |
|---|---|---|---|
| Writer fps     | `_anim_helpers.py::save_anim` (`gif_speed`/`mp4_speed`) | 1/3 / 1/4 | Wall-clock playback rate. GIF plays at 1/3× the natural fps, MP4 at 1/4×. Same frames; longer per-frame duration. |
| `--advance-s`  | each `make_*_anim.py` | 0.125 (peak), 0.5 (FFT/RR) | Signal-time advance per frame. *True slow motion at the same playback fps* — halves the on-screen signal traversal rate without changing wall-clock duration. |

Both knobs slow what the viewer perceives, in different ways. The writer-fps
knob extends wall-clock duration (each frame held longer). The advance-s knob
keeps duration constant but slows the on-screen signal traversal. The
defaults stack both for a 32-second autoplay loop where each MP4 frame
advances 0.125 s of signal at 7.5 fps playback ≈ 0.94× real time — the
signal flows past the plot at essentially the firmware's actual rate.

To get a faster slide back: pass `--advance-s 0.25` (peak) or `--advance-s
1.0` (FFT/RR) on the command line. To shorten the wall-clock loop too, edit
`save_anim`'s `gif_speed` / `mp4_speed` toward 1.0 in `src/_anim_helpers.py`.

All loop seamlessly. The peak animation shows: raw PPG (top), Q15 band-passed
signal with detected peaks (middle), live HR readout (bottom). The FFT
animation shows: raw + band-passed overlaid, the 256-pt FFT magnitude² across
the HR band (with the chosen bin in red and the sub-harmonic candidate in
amber when the swap fires), live HR. The RR animation shows: raw + lowpass
overlaid, the 24-bin Goertzel scan, live RR.

## Regenerating

```bash
# Peak detector (240 frames @ 30 fps)
uv run python src/make_peak_anim.py

# FFT spectral path with sub-harmonic check (default α = 1/3, DIVISOR=3)
uv run python src/make_fft_anim.py
uv run python src/make_fft_anim.py --subharmonic-divisor 2   # looser α=0.5

# Respiration rate (BW path)
uv run python src/make_rr_anim.py

# Smaller GIF for any:
uv run python src/make_<peak|fft|rr>_anim.py --frames 180 --fps 20

# Use a different BIDMC record:
uv run python src/make_<peak|fft|rr>_anim.py --record bidmc02
```

Common CLI flags across all three: `--record`, `--cache-dir`, `--window-s`
(analysis window for FFT/RR), `--advance-s`, `--frames`, `--fps`, `--out-dir`.
FFT-specific: `--fft-n`, `--fft-decim`, `--subharmonic-divisor`. RR-specific:
`--lp-taps`, `--lp-cutoff`, `--decim`.

## Why no cycle-count / FPS-on-target overlay

Same honesty constraint as the rest of the project: QEMU `lm3s6965evb` is a
functional emulator, not cycle-accurate. The HR number shown is the actual
firmware computation; any "this runs at X kHz on the target" overlay would be
fiction without a real-hardware or Renode measurement (see Future Work in the
top-level README).
